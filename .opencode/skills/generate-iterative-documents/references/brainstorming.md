# 需求澄清与头脑风暴

## 提问原则

- 本文各阶段描述的是需要覆盖的信息目标，不是必须逐条照读的固定问卷。
- 每轮先把用户回答归纳成 1–3 条简短结论，再只问一个最影响后续文档的问题。
- 每轮归纳后立即写入项目 memory；下次提问前先检索相似共性问题，命中时只请用户确认或修正候选答案。
- 题干使用一句短句；选项根据当前上下文动态生成 2–3 个互斥答案。
- 框架会自动提供自由输入入口，不要再添加“用户输入”“其他”或“自定义”选项。
- “不涉及”“当前不清楚”“继承既有需求”仅在语义适用时出现，不作为每题固定选项。
- 对模糊回答，先结合已知上下文给出你的理解并请用户确认；仅在必要时围绕“触发条件—输入—处理—输出—失败表现—依赖—约束—验证方法”追问。
- 区分事实、假设和待确认项。不要替用户发明接口名、参数或芯片能力。
- 写入字段前进行提炼：删除聊天式表达和重复内容，保留可追踪的事实、约束与判定标准；不能直接粘贴用户原话充当整理结果。
- 提问的目标不是“收齐字段”，而是构建一个开发者可实现、测试人员可判定的需求模型。每个结论都应能回答它影响哪条流程、哪个状态、哪个输出或哪个验收点。

## SRS、SD 与伪代码的职责

- **SRS 是需求基线**：描述系统对外必须表现出的能力和约束，即“谁在什么条件下提供什么输入，系统必须产生什么可观察结果；异常和边界时必须怎样表现”。它应可验收，但不规定内部函数如何拆分。
- **SD 是实现基线**：描述软件为了满足 SRS 将如何组织模块、函数、数据、状态、调用顺序、错误转换、并发控制和资源生命周期。每项关键设计都应能追溯到一条 SRS 行为或约束。
- **伪代码是理解校验**：不是额外发明实现，而是把已确认的 SD 主流程、关键分支和清理路径写成可推演结构。若无法在不猜测的情况下写出伪代码，说明需求尚未理解到位，必须继续向开发者确认。
- 同一事实不要原样复制到 SRS 和 SD。SRS 写可观察契约，SD 写实现该契约的设计。例如，SRS 写“芯片调用失败时返回映射后的错误码且不改变设备状态”，SD 再写错误码映射点、状态提交时机和回滚/释放步骤。

## 第一阶段：基本信息

1. 迭代名称（例如 `B240迭代`）及英文中使用的迭代标识（例如 `B240`）。
2. 作者、修订版本、修订日期。
3. AR 列表；每次只处理一个 AR，询问是否还有下一个。

## 第二阶段：逐 AR 澄清

对每个 AR 按缺口选择性确认，用户材料已覆盖的内容直接归纳，不重复提问：

1. 功能需求标题、AR 号、新功能或继承需求。
2. 背景/用户价值：为什么需要、当前问题、受影响角色或模块。
3. 输入与 API：接口签名、参数、类型、单位、合法范围、缺省值；驱动与 LCNE、芯片的边界。
4. 处理：前置检查、调用链、状态变化、同步/异步、并发、超时、重试、资源回收。
5. 输出：返回值、输出参数、日志/事件、状态、副作用，以及可观测的成功标准。
6. 异常：非法值、空值、边界值、依赖不可用、芯片失败、重复调用、并发冲突及错误码。
7. 外部依赖：ko、固件、芯片版本、其他接口、配置、服务及初始化顺序。
8. 场景/设计约束：芯片版本，debug/release，权限，前置操作，兼容性与继承行为。
9. 性能：时延、吞吐、并发、CPU/内存、满规格及时序要求。
10. 质量：可靠性、安全性、兼容性、资源泄漏、恢复及日志要求。
11. 可测试性：dt 函数名、可注入异常、观测点、mock/stub、自动化限制。
12. 是否需要接口表、参数表、流程图、时序图或状态图；需要时把提醒写入相关正文。

### 逐 AR 的推演与确认

完成一轮事实收集后，不立即切换到下一个 AR，而是按以下顺序推演：

1. 从触发条件和前置状态开始，使用一个正常输入逐步推演到返回值、输出参数、状态变化和副作用。
2. 对每个校验、外部调用、状态提交和资源申请点，确认失败时的返回、日志、回滚与释放。
3. 对边界值、重复调用、并发、超时和异步完成逐项判断是否会改变主流程。
4. 检查调用者是否能仅凭 SRS 判断成功或失败，开发者是否能仅凭 SD 确定职责、顺序和清理路径。
5. 形成并展示以下伪代码骨架；只保留适用于当前 AR 的部分：

```text
function <已确认接口或明确占位符>(input):
    validate preconditions and input
    if validation fails:
        return confirmed error/result

    acquire or initialize required resources
    call confirmed dependency
    if dependency fails:
        rollback state if needed
        release resources
        return mapped error/result

    update or commit state at the confirmed point
    publish output/event/log if required
    release temporary resources
    return success result
```

6. 将推演结果拆成“SRS 候选内容、SD 候选内容、伪代码、待确认项”，请开发者纠正。开发者确认后才进入下一个 AR。

以下情况说明仍需继续提问：伪代码中出现无法命名的关键输入或依赖；分支只有“处理失败”而无确定结果；状态何时提交不明确；资源由谁释放不明确；并发调用结果不明确；SRS 验收标准只能写成“功能正常”。

## 第三阶段：文档粒度确认与反向总结

所有 AR 完成后，请用户确认以下汇总：

- SRS：目的、范围、总体概述、软件功能、设计约束、性能、质量、可测试性。
- SD：目的、范围、软件功能。
- 对每个 AR 并排回读 SRS 与 SD：确认 SRS 没有泄漏不必要的内部实现，SD 没有停留在“调用接口完成处理”这类无法指导编码的空泛表述。
- 询问开发者哪些接口签名、参数表、状态机、流程图、时序图、错误码表和伪代码应正式呈现在 SRS 或 SD；按照模板可容纳的位置写入或添加明确的插图/表格提示。
- 新功能需求与继承需求分别列出；范围中明确“不涉及其他 UDK 需求”。
- 修订记录的日期、版本、AR、章节、描述、作者。

最终确认时应明确展示：

1. SRS 中将承诺的外部行为和验收标准；
2. SD 中将固化的实现决策，以及仍允许开发者自行决定的实现细节；
3. 伪代码是否覆盖正常、异常、回滚/清理和关键并发路径；
4. 所有“当前不清楚”、继承来源和可能阻塞编码的未决项。

## 第四阶段：STC 用例设计

逐接口建立输入等价类和边界表，再选择可构造的用例类型：

- 基本：功能、异常、性能、安全、可靠性、兼容性。
- 定制：满规格、时序、资源回收、并发、边界校验。

每条用例必须具备测试项、标题、H/M/L、预置条件、抽象输入、可执行命令和可判定结果。操作步骤格式优先为：

`./udk_dt_tool <dt_func_name> <parameters...>`

若 dt 函数名或参数未确定，继续询问，不生成伪命令。预置条件默认选项为 A“无条件”或 B“mami已纳管设备”。功能用例覆盖正常返回和输出；异常用例覆盖明确错误码/报错退出。用例编号由需求标题的稳定缩写与三位序号组成，避免仅用自然语言标题导致重复。
