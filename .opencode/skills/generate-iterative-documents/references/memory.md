# 项目级 Memory

Memory 用于减少重复提问。它保存经过归纳的候选结论，但不能替代用户对当前文档的确认。

## 存储位置

- 默认文件：当前工作项目根目录下的 `.iterative-documents/memory.json`。
- 不把运行时 memory 写入 skill 的 `references/` 或 `assets/`。
- 若项目已有明确的记忆文件位置，沿用项目约定。

建议结构：

```json
{
  "version": 1,
  "entries": [
    {
      "id": "author-default",
      "topic": "文档作者",
      "scope": {
        "project": "UnionSV110",
        "iteration": "*",
        "ar": "*"
      },
      "keywords": ["作者", "修订记录", "author"],
      "question_intent": "本次文档作者是谁",
      "answer": "张三",
      "status": "confirmed",
      "source": "user",
      "updated_at": "2026-06-15"
    }
  ]
}
```

## 每轮写入

收到用户每次回答后执行一次 upsert：

1. 结合当前问题和上下文，把回答归纳为一条可复用结论。
2. 标记适用范围：`project`、`iteration`、`ar`；`*` 仅用于用户明确表示长期通用的信息。
3. 用户明确确认的内容使用 `confirmed`；Agent 推断但尚未确认的内容使用 `candidate`；“当前不清楚”使用 `unknown`。
4. 用“问题意图 + 适用范围”查找已有条目。相同事实更新 `answer`、`source` 和 `updated_at`，不要新增近义副本。
5. 一次回答包含多个独立事实时可以拆成多条 entry，但每条只表达一个事实。
6. 不保存聊天原文；只保存归纳后的答案和必要来源。不要记录与文档生成无关的敏感信息。

## 检索与确认

提问前按以下顺序检索：

1. 当前 AR 完全匹配。
2. 当前迭代匹配、AR 为 `*`。
3. 当前项目匹配、迭代和 AR 为 `*`。

只有 `confirmed` 条目可以作为优先候选。匹配时综合比较 `question_intent`、`topic`、`keywords` 和适用范围，不要只按字面关键词命中。

命中可靠候选后，使用简短确认题：

- 题干：`沿用“预置条件：mami已纳管设备”吗？`
- 选项：`确认沿用`、`需要修改`

框架会自动添加自由输入入口，不要提供“其他”或“自定义”选项。用户确认后，把候选写入当前字段并更新 memory 的时间和当前适用范围；用户修改后，以新结论 upsert。

## 冲突与失效

- 当前用户输入、正式模板和当前需求材料始终高于 memory。
- 多条候选冲突时，不自行选择；用一句话展示最相关差异并请用户确认。
- 接口签名、参数范围、错误码、芯片版本、性能指标等易变化事实必须匹配当前 AR，且每个新 AR 都要再次确认。
- 用户否定旧结论时更新或标记旧条目为 `superseded`，后续不再作为候选。
- memory 文件损坏或结构不合法时先备份，再创建空结构；不要凭损坏内容继续推断。
